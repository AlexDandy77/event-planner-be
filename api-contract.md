# Event Planner API Contract

## Purpose

This document is the concrete HTTP contract between the React frontend and the .NET backend.

Use this file for:

- endpoint paths
- query parameters
- request body schemas
- response body schemas
- validation rules
- error response shapes
- frontend TypeScript API types
- backend Application-layer DTO names

## Important Separation

There are three related but different concepts:

## 1. Domain / EF Entities

These are internal backend C# classes.

They represent business and persistence state inside the backend. EF Core maps these classes to database tables.

Examples:

- `CalendarEvent`
- `User`

Rules:

- They live in `EventPlanner.Domain`.
- EF Core configuration lives in `EventPlanner.Infrastructure`.
- They use C# naming such as `StartDateTime`, `PasswordHash`, `IsDeleted`.
- They can contain internal fields that the frontend must never see.
- Controllers should not return them directly.

## 2. API Resources / Response DTOs

These are the JSON shapes returned to the frontend.

In code, these should be response DTOs, not EF entities.

Examples:

- `EventResponse`
- `UserResponse`
- `AuthResponse`

Rules:

- They live in `EventPlanner.Application`.
- They use JSON `camelCase`.
- They expose only safe frontend fields.
- They can be similar to EF entities, but they are not the same thing.
- Sensitive fields such as `passwordHash` are never returned.

## 3. Request DTOs / Query DTOs

These are the shapes the frontend sends to the backend.

Examples:

- `CreateEventRequest`
- `UpdateEventRequest`
- `RegisterRequest`
- `LoginRequest`
- `EventQueryParameters`

Rules:

- They live in `EventPlanner.Application`.
- They include only fields the client is allowed to provide.
- They do not include backend-owned fields such as `id`, `createdAt`, `updatedAt`, or `isDeleted`.
- They are validated before the Application service executes the use case.

## Contract Version

```txt
Version: v1
Base path: /api
Current auth mode: public during initial CRUD
Future auth mode: JWT protected for authenticated user flows
```

## General API Rules

- API returns JSON.
- JSON field names use `camelCase`.
- IDs are UUID strings.
- Date-time values use ISO 8601 UTC strings.
- Backend stores and returns `DateTimeOffset` values.
- Frontend sends only editable fields in create/update requests.
- Backend owns generated fields:
  - `id`
  - `organizerId`
  - `createdAt`
  - `updatedAt`
  - `isDeleted`
- Events are soft-deleted with `isDeleted = true`.
- Deleted events are excluded from regular event list responses.
- Error responses use a stable Problem Details style shape.

## Internal Entity Reference

This section describes internal backend models. It is not the HTTP response contract.

## CalendarEvent Entity

```txt
Id: Guid
Title: string
Description: string?
Category: EventCategory
StartDateTime: DateTimeOffset
EndDateTime: DateTimeOffset
OrganizerId: Guid
CreatedAt: DateTimeOffset
UpdatedAt: DateTimeOffset
Color: string
IsDeleted: bool
```

Domain rules:

- Title is required.
- Start date must be before end date.
- Color must be a valid hex color.
- Organizer must exist.
- Deleted events are soft-deleted through `IsDeleted`.

## User Entity

```txt
Id: Guid
FirstName: string
LastName: string
Email: string
Role: UserRole
Phone: string?
PasswordHash: string
IsActive: bool
CreatedAt: DateTimeOffset
UpdatedAt: DateTimeOffset
```

Domain rules:

- Email is unique.
- Password hash is never returned from the API.
- Inactive users cannot log in.
- Role controls authorization.

## Response DTOs

## EventResponse

Returned by event endpoints.

```json
{
  "id": "4c516b2a-7f1f-4c19-9a0d-09362d3d2fd1",
  "title": "Tech Career Fair 2026",
  "description": "An event for IT students and companies.",
  "category": "online",
  "startDateTime": "2026-05-12T09:00:00Z",
  "endDateTime": "2026-05-12T17:00:00Z",
  "organizerId": "8a5edca7-8f80-4ed6-8b91-01863782b51b",
  "createdAt": "2026-04-07T10:00:00Z",
  "updatedAt": "2026-04-07T10:00:00Z",
  "color": "#123123",
  "isDeleted": false
}
```

| Field | Type | Notes |
| --- | --- | --- |
| `id` | `string` | UUID generated by backend. |
| `title` | `string` | Event title. |
| `description` | `string \| null` | Event description. |
| `category` | `string` | Supported event category value. |
| `startDateTime` | `string` | ISO 8601 UTC date-time. |
| `endDateTime` | `string` | ISO 8601 UTC date-time. |
| `organizerId` | `string` | Development organizer first, current user after auth. |
| `createdAt` | `string` | Set by backend. |
| `updatedAt` | `string` | Set by backend. |
| `color` | `string` | Hex color such as `#123123`. |
| `isDeleted` | `boolean` | `false` for regular results. |

## Query DTOs

## EventQueryParameters

Used by the event list endpoint.

| Parameter | Type | Required | Default | Notes |
| --- | --- | --- | --- | --- |
| `from` | `string` | no | none | ISO 8601 UTC date-time. Includes events ending after this value. |
| `to` | `string` | no | none | ISO 8601 UTC date-time. Includes events starting before this value. |
| `category` | `string` | no | all | Must match supported event category values. |
| `search` | `string` | no | none | Searches title and description. |
| `sortBy` | `string` | no | `startDateTime` | `startDateTime`, `title`, or `createdAt`. |
| `sortDirection` | `string` | no | `asc` | `asc` or `desc`. |

Example:

```http
GET /api/events?from=2026-05-18T00:00:00Z&to=2026-05-25T00:00:00Z&sortBy=startDateTime&sortDirection=asc
```

## Event Endpoints

## GET /api/events

Returns events visible in the requested range.

Default behavior:

- Exclude soft-deleted events.
- Sort by `startDateTime asc`.
- When auth is enabled, return events visible to the current user.

Success:

```http
200 OK
```

```json
[
  {
    "id": "4c516b2a-7f1f-4c19-9a0d-09362d3d2fd1",
    "title": "Tech Career Fair 2026",
    "description": "An event for IT students and companies.",
    "category": "online",
    "startDateTime": "2026-05-20T09:00:00Z",
    "endDateTime": "2026-05-20T11:00:00Z",
    "organizerId": "8a5edca7-8f80-4ed6-8b91-01863782b51b",
    "createdAt": "2026-05-01T10:00:00Z",
    "updatedAt": "2026-05-01T10:00:00Z",
    "color": "#DC2626",
    "isDeleted": false
  }
]
```

Errors:

```txt
400 Bad Request - invalid query parameter
401 Unauthorized - future auth mode only
```

## GET /api/events/{id}

Returns one event by ID.

Success:

```http
200 OK
```

Response: `EventResponse`

Errors:

```txt
400 Bad Request - invalid UUID
404 Not Found - event does not exist or is deleted
401 Unauthorized - future auth mode only
```

## POST /api/events

Creates an event.

During initial CRUD, `organizerId` can be assigned from a seeded development organizer. After JWT auth is added, the backend must assign it from the current authenticated user.

Request: `CreateEventRequest`

Success:

```http
201 Created
Location: /api/events/4c516b2a-7f1f-4c19-9a0d-09362d3d2fd1
```

Response: `EventResponse`

Errors:

```txt
400 Bad Request - invalid request body
401 Unauthorized - future auth mode only
```

## PUT /api/events/{id}

Updates an event.

Request: `UpdateEventRequest`

Success:

```http
200 OK
```

Response: `EventResponse`

Errors:

```txt
400 Bad Request - invalid UUID or request body
401 Unauthorized - future auth mode only
403 Forbidden - future auth mode, user cannot update this event
404 Not Found - event does not exist or is deleted
```

## DELETE /api/events/{id}

Soft-deletes an event.

Success:

```http
204 No Content
```

Errors:

```txt
400 Bad Request - invalid UUID
401 Unauthorized - future auth mode only
403 Forbidden - future auth mode, user cannot delete this event
404 Not Found - event does not exist or is already deleted
```

## Auth Endpoints

Auth is planned for a later backend phase, but the contract is defined now so the frontend can prepare types and services.

## POST /api/auth/register

Creates a user account.

Request: `RegisterRequest`

Success:

```http
201 Created
```

Response: `AuthResponse`

Errors:

```txt
400 Bad Request - invalid request body
409 Conflict - email already exists
```

## POST /api/auth/login

Authenticates an existing user.

Request: `LoginRequest`

Success:

```http
200 OK
```

Response: `AuthResponse`

Errors:

```txt
400 Bad Request - invalid request body
401 Unauthorized - invalid credentials or inactive user
```

## GET /api/auth/me

Returns the current authenticated user.

Headers:

```http
Authorization: Bearer jwt-token
```

Success:

```http
200 OK
```

Response: `UserResponse`

Errors:

```txt
401 Unauthorized - missing, invalid, or expired token
```

## Validation Rules

## CreateEventRequest And UpdateEventRequest

| Field | Rule |
| --- | --- |
| `title` | Required, 1 to 120 characters. |
| `description` | Optional, max 1000 characters. |
| `category` | Required, must be supported category. |
| `startDateTime` | Required, valid ISO date-time. |
| `endDateTime` | Required, valid ISO date-time, later than `startDateTime`. |
| `color` | Required, valid hex color in `#RRGGBB` format. |

## RegisterRequest

| Field | Rule |
| --- | --- |
| `firstName` | Required, 1 to 80 characters. |
| `lastName` | Required, 1 to 80 characters. |
| `email` | Required, valid email, unique. |
| `password` | Required, minimum 8 characters. |
| `phone` | Optional, max 30 characters. |

## LoginRequest

| Field | Rule |
| --- | --- |
| `email` | Required, valid email. |
| `password` | Required. |

## Error Response Shape

Use this shape for validation and stable business errors.

```json
{
  "type": "https://httpstatuses.com/400",
  "title": "Validation failed",
  "status": 400,
  "detail": "One or more validation errors occurred.",
  "errors": {
    "title": ["Title is required."],
    "endDateTime": ["End date must be later than start date."]
  }
}
```

Use this shape for simple not found or conflict errors.

```json
{
  "type": "https://httpstatuses.com/404",
  "title": "Event not found",
  "status": 404,
  "detail": "Event with the requested id was not found."
}
```

## Contract Acceptance Criteria

- Internal EF/domain entities are not returned directly by controllers.
- Response DTOs expose only frontend-safe fields.
- Create/update payloads do not include backend-owned fields.
- Event CRUD endpoints are named and documented.
- Filtering and sorting query params are documented.
- Auth endpoints are named and documented for future work.
- Validation and error response shapes are stable.
- Frontend TypeScript types can be created from this contract.
- Backend DTOs and controllers can be implemented from this contract.
